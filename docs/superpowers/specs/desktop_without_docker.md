不用docker可以吗:
可以，但要接受一个现实：你们现在这套「乙（本机跑任务）」之所以适合 Docker，是因为它把 MySQL + Redis + FastAPI + 独立下载服务 + R/Python 运行时 + code/ 脚本 这些重依赖“一次性封装”了。不用 Docker 当然能做，只是交付与维护方式会变。

不用 Docker 的两条主路（给你选方向）
方案 A：原生安装包（推荐的“无 Docker 全功能”路径）
做法：给 Win/Linux 做安装器，把 Python 运行环境、依赖、R（以及 R 包）、MySQL/MariaDB、Redis（可选）都装好；桌面壳启动本机服务进程（8005/8001），前端用本地静态服务/内嵌 WebView 打开 127.0.0.1。
优点：用户体验像传统软件，不要求 Docker。
缺点：你们要承担 跨 OS 打包、依赖冲突、R 环境、数据库安装/迁移、升级/卸载 的复杂度；总体工作量通常 显著高于 Docker。
方案 B：内置“便携运行时”（更像自带一套运行环境）
做法：在安装目录里自带 Python+依赖（例如打成可执行/便携环境），R 也打包或随包提供；数据库尽量改为 SQLite 或“嵌入式/便携版数据库”（如果你们业务允许）。
优点：对系统侵入更小，安装更简单。
缺点：如果你们必须 MySQL（而且已有不少 MySQL 交互/表结构），那就很难“便携化”；而且 R + 大依赖 仍然会让包很大、兼容性要做很多测试。


不想维护 Docker 镜像/仓库：
如果你的核心诉求是 “不想维护 Docker 镜像/仓库”，那桌面离线全功能依然可做，但主线会从「维护镜像」转成「维护安装包/运行时」，通常有三种落地方式：

方式 1（最少维护，偏“自助环境”）：你只发布 桌面壳 + 配置/启动脚本，要求用户机器上自行具备 Python/Conda + R + MySQL/Redis（或你们提供一键安装脚本）。

优点：你们发布物很轻，不维护镜像仓库
缺点：用户环境差异导致问题更多，支持成本上升
方式 2（维护安装包，不维护镜像）：你们做 Windows/Linux 安装包，把需要的运行时（Python/R/依赖、数据库等）一起装好，桌面壳只负责启动本机服务。

优点：用户体验最好、可离线
缺点：你们要维护 安装器、升级、卸载、跨 OS 兼容（工作量通常不小）
方式 3（降低数据库依赖）：尽量把本机数据库从 MySQL 迁到 SQLite（或将部分状态改为文件/嵌入式存储），减少“系统级依赖”，让安装包更可控。

优点：大幅减少安装复杂度
缺点：需要改后端数据层与迁移策略（影响面最大）
接下来只问一个关键问题，决定你更适合走哪条：

你能接受让用户自行安装/准备 Conda+R+MySQL（你们提供脚本指导即可），还是必须做到“下载安装到手就能用、无需用户装依赖”？（二选一回答即可）

# 桌面版（乙）：无 Docker、免依赖安装方案（原生安装包）

日期：2026-04-01  
状态：草案（待产品决策）  
范围：面向 Windows + Linux 用户，提供“下载安装到手就能用、无需用户安装依赖”的桌面版；任务在用户本机执行；不依赖 Docker，也不需要维护镜像仓库。

## 方案优缺点与与 Docker 两方案对比

### 本方案（无 Docker 原生安装包）优点

- **用户门槛低**：无需安装 Docker、Conda、R、MySQL/Redis 等；安装完成即可用。
- **首装可离线**：可通过离线安装介质交付（取决于你们是否把运行时/依赖随包带齐）。
- **运维入口统一**：用户只关心“应用是否在跑”，不需要理解容器、镜像、卷等概念。

### 本方案（无 Docker 原生安装包）缺点

- **维护重心转移**：不维护镜像仓库 ≠ 不维护依赖；你们要维护 Win/Linux 安装与升级、卸载、服务注册、权限与日志采集。
- **兼容与测试成本更高**：R 与其包、Python 科学计算栈、数据库服务在不同 OS/发行版上的兼容性更难控。
- **安全面更大**：本机安装数据库/服务进程需要处理端口暴露、数据目录 ACL、密钥存放、自更新签名等。

### 与两种 Docker 方案对比（摘要）

| 维度 | 无 Docker 原生安装包（本文档） | 联网 pull Docker | 完全离线 tar Docker |
|------|-------------------------------|------------------|----------------------|
| **是否要维护镜像仓库** | 否 | 是 | 可选（离线交付可弱化仓库） |
| **用户是否需装 Docker** | 否 | 是 | 是 |
| **首装离线** | 可 | 不可 | 可 |
| **升级方式** | 安装包/自更新（你们维护） | pull 镜像 + 重启栈 | 新离线包/新 tar |
| **环境一致性** | 中 | 高 | 高 |
| **总体复杂度** | **中高** | 中 | 中高 |

## 1. 背景与目标

### 1.1 背景

当前系统属于“乙（本机跑任务）”，涉及：

- Vue 前端（Vite 构建产物）
- FastAPI 主服务（8005）与独立下载服务（8001）
- MySQL（任务/文件/用户等数据）
- Redis（如用于队列/缓存/会话）
- Rscript + Python 脚本（位于 `code/`，并涉及较多依赖）

Docker 方案的优势是把这些依赖封装在镜像里；无 Docker 方案要实现“免依赖安装”，就必须把运行时与依赖在安装阶段交付到用户机器上。

### 1.2 成功标准

- 用户下载并安装桌面版后，无需额外安装任何依赖即可运行。
- 支持 Windows 与 Linux（首版可设优先级，但最终两者都要可用）。
- 支持离线使用：不依赖外部服务器即可完成核心流程（上传/处理/训练/下载结果）。
- 升级可控：版本升级不要求用户手动处理数据库/依赖。

---

## 2. 总体架构

### 2.1 桌面壳

Electron 或 Tauri 作为 GUI 外壳，职责：

- 管理本机后台服务的启动/停止/重启
- 展示运行状态、端口、健康检查、关键日志入口
- 打开内置 WebView 指向本地入口（如 `http://127.0.0.1:<uiPort>`）

### 2.2 本机服务进程

建议拆成三个进程（同一发行包内）：

- **UI 静态服务**：提供 Vue 构建产物（nginx 或轻量静态服务器）。
- **FastAPI 主服务**：提供 `/api`、`/ws`。
- **下载服务**：提供结果/示例下载；推荐改为 **同源路径**（由 UI 服务反代到下载服务），避免前端硬编码 `localhost:8001`。

---

## 3. 运行时与依赖交付（实现阶段的关键决策点）

为满足“无需用户装依赖”，必须明确如何携带：

- **Python 运行时与依赖**：嵌入式 Python / 便携环境 / 打成服务可执行文件。
- **R 运行时与 R 包**：随包提供 R 与必要 CRAN/Bioconductor 包缓存，确保离线可运行。
- **数据库**：若坚持 MySQL（现状），安装包需安装并配置 MySQL/MariaDB 服务，并处理初始化与迁移；若允许改造，可评估 SQLite 以降低系统级依赖。
- **Redis**：如确实需要，则随包提供并管理其服务进程（或替代机制）。

---

## 4. 数据目录与路径规范

- 明确默认工作目录（建议在用户目录下的应用数据目录，而非安装目录）。
- 统一 `code/`、`uploaded_files/`、`download_data/`、日志目录的映射方式，避免业务代码写死开发机绝对路径。

---

## 5. 安装、升级、卸载策略

- **安装**：写入运行时与依赖，创建数据目录与初始配置，初始化数据库。
- **升级**：前端/后端/R 包升级的兼容策略；数据库 schema 迁移策略（自动迁移、失败保护）。
- **卸载**：是否保留用户数据（默认保留并提示用户选择）。

---

## 6. 测试与验收

- Windows：干净机器（无 Python/R/MySQL/Redis）冷启动安装→跑通最小任务→下载结果。
- Linux：至少 1~2 个主流发行版做同样路径；覆盖 systemd 权限与端口占用。
- 离线：断网情况下完成首装与运行（若选择“离线包带齐依赖”）。
